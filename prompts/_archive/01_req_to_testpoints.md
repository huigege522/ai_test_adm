# 阶段一：需求文档 → 测试点提取

**全景位置**：整条测试链路（含 **UI E2E**）见 [00_README.md](00_README.md)。

**使用场景**：拿到 PRD / 需求文档后，第一步让 AI 结构化提取所有测试点。  
**在 Cursor 中使用**：打开需求文档，直接粘贴 Prompt + 需求内容发送。

---

## Prompt 模板

```
你是一名资深测试工程师，请阅读以下需求文档，按以下四个维度输出结构化测试点清单。

输出格式要求：
- 每条测试点包含：编号、维度、模块、测试点描述、优先级（P0/P1/P2）
- 用 Markdown 表格输出
- P0 = 核心流程不可缺失，P1 = 重要场景，P2 = 优化/边缘场景

四个维度：
1. 【功能测试点】正向业务流程，"用户能正常完成XXX"
2. 【边界值测试点】数值上下限、字符串长度、列表为空/满
3. 【异常测试点】错误输入、缺失必填、状态不允许、越权访问
4. 【业务规则测试点】约束条件、状态机转换、并发冲突

需求文档：
---
[在此粘贴需求文档内容]
---

输出示例（按此格式）：
| 编号 | 维度 | 模块 | 测试点描述 | 优先级 |
|------|------|------|-----------|--------|
| TC001 | 功能 | 用户注册 | 填写完整信息可成功注册 | P0 |
```

---

## 进阶：需求 vs 代码交叉验证

完成上述输出后，在 Cursor 中 @ 代码文件，追问：

```
对照以上测试点清单，阅读 @[Controller文件] 和 @[Service文件]，找出：
1. 需求中有但代码未实现的功能（漏实现）
2. 代码中有但需求未提及的逻辑分支（隐藏逻辑）
3. 可能存在的边界条件未处理问题

每项以 "⚠️ 风险" 开头，说明风险描述和建议验证方式。
```

---

## 输出产物

将 AI 输出的清单保存为：`test_data/testpoints/[模块目录]/testpoints_[模块显示名].md`（模块目录与 `tests/api/{模块目录}/` 对齐，如 `basic_info`），并标注生成日期和需求版本号。

## 与后续阶段衔接

- **阶段二～四（API、数据、断言）**：用本清单的 **功能 / 边界 / 异常 / 规则** 全面覆盖接口与数据库断言。
