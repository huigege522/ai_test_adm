# 阶段二-B：代码库 → 逻辑分支测试用例生成

**全景位置**：[00_README.md](00_README.md)。本阶段仍面向 **`tests/api/`** 与请求构造；**前端分支与页面状态**不在此覆盖，若需验证请走 **阶段五 [06_e2e_ui_flow.md](06_e2e_ui_flow.md)**。

**使用场景**：有代码阅读权限时，让 AI 分析代码逻辑补充接口文档未体现的分支场景。  
**在 Cursor 中使用**：@ 具体代码文件后使用 Prompt。

---

## Prompt 模板

### 步骤 1：分析代码逻辑分支

```
阅读以下代码文件：
@[Service 层文件]  @[Controller 层文件]

请完成：
1. 列出所有 if/else、switch、try/catch 分支，以树状结构展示
2. 标注每个分支的触发条件（什么参数/状态触发）
3. 标注是否有现有测试用例覆盖（参考 tests/api/[模块名]/test_*.py）
4. 对未覆盖的分支，标注 "⚠️ 未覆盖"

输出格式：
- [方法名]
  ├── 分支1: [触发条件] → [结果] [已覆盖/⚠️ 未覆盖]
  └── 分支2: [触发条件] → [结果] [已覆盖/⚠️ 未覆盖]
```

### 步骤 2：为未覆盖分支生成用例

```
针对上一步标注的 "⚠️ 未覆盖" 分支，生成对应的 pytest 测试函数：

要求：
- 使用 ApiClient 发送请求触发该分支
- 说明如何构造触发该分支的请求参数/前置数据
- 如需特殊数据库状态，在 fixture 或测试函数开头写明 SQL 前置条件（注释形式）
- 函数命名体现触发条件，如 def test_库存为零时下单失败(self)
```

---

## 针对复杂业务逻辑（状态机）

```
阅读 @[Service 文件]，该模块存在状态机逻辑。

请：
1. 绘制状态转换图（Mermaid 格式）
2. 列出所有合法状态转换路径
3. 列出所有非法状态转换（应被拒绝的）
4. 为每条非法转换生成一个 pytest 负向测试用例

合法转换 → 生成 smoke 标记用例
非法转换 → 生成 negative 标记用例
```

---

## 输出产物

追加至：`tests/api/[模块名]/test_*.py` 或新建 `tests/api/[模块名]/test_*_branch.py`

与 **UI E2E** 的分工：分支结论若体现为「仅 UI 可见的错误提示」，在测试点中标注优先级，并交由阶段五用 Playwright 断言可见文案或禁用态。
