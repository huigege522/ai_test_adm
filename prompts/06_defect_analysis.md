# 阶段五：缺陷分析 & 回归建议

**使用场景**：
  A. 测试失败时，快速定位根因
  B. 需求/接口变更时，识别影响范围，生成回归建议

---

## Prompt 模板 A：测试失败根因分析

```
以下 pytest 测试用例执行失败，请帮我分析根因：

==== 失败信息 ====
[粘贴 pytest 输出，包含 FAILED 行和 AssertionError 详情]

==== 接口定义 ====
@[Apifox 接口文档 或 粘贴接口定义]

==== 相关代码 ====
@[Controller 文件]  @[Service 文件]

==== 数据库表结构 ====
[粘贴相关 CREATE TABLE]

==== 请分析 ====
1. 最可能的失败原因（代码 Bug / 环境问题 / 测试用例本身错误 / 数据问题）
2. 需要检查的代码位置（精确到方法名和行号）
3. 需要验证的数据库状态（给出 SQL 查询语句）
4. 建议的修复方向
5. 如果是测试用例本身写错了，给出修正后的代码
```

---

## Prompt 模板 B：接口变更影响分析

```
Apifox 中以下接口发生了变更，请分析测试用例的影响范围：

==== 变更描述 ====
接口：[POST /api/user/create]
变更内容：
- 新增必填参数 phone（手机号，string，11位）
- 返回体 data 中新增字段 userId（string）
- 原 email 参数改为非必填

==== 现有测试文件 ====
@[tests/api/test_user.py]  @[tests/integration/test_flow_example.py]

==== 请输出 ====
1. 【需修改的用例】列出文件名 + 函数名 + 需修改原因
2. 【需新增的用例】列出新场景及其测试点描述
3. 【可删除的用例】列出已无效的用例
4. 直接给出修改后的代码（diff 格式）
```

## Prompt 模板 C：需求变更回归范围识别

```
需求文档发生了以下变更（版本 v1.2 → v1.3）：
[描述变更内容，或粘贴变更日志]

对照代码库 @[相关 Service/Controller 文件] 和测试文件 @[tests/] 目录，请：

1. 识别哪些接口/模块受到影响
2. 列出受影响的现有测试用例（文件名 + 函数名）
3. 生成回归测试清单（P0 必须回归 / P1 建议回归）
4. 标出可能引入的新风险点（需要新增测试覆盖的场景）

输出格式：
## 影响接口
- [接口名] [变更类型：参数变更/行为变更/新增/废弃]

## 受影响用例
| 文件 | 函数名 | 处理方式 |
|------|--------|---------|

## 新增测试点
| 测试点描述 | 优先级 | 建议文件位置 |
```

## 日常缺陷记录模板

建议在 `test_data/` 目录下维护 `bug_tracker.md` 记录 AI 辅助分析的缺陷：

```markdown
## Bug #001
- 发现时间：YYYY-MM-DD
- 失败用例：tests/api/test_user.py::TestUserCreate::test_正常创建用户
- AI 分析根因：UserService.create() 未验证 phone 格式
- 缺陷定位：src/service/UserService.java:L45
- 修复状态：待修复 / 已修复(PR#xxx) / 转需求
```
